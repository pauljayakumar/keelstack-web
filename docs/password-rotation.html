Password.
Rotation. Guide.

How often should I rotate?

tl;dr Modern guidance (NIST SP 800-63B, 2017+) doesn't recommend time-based rotation for strong passwords. It encourages weaker passwords (people just bump a number at the end). Rotate when something happens — not on a calendar.

Rotate immediately when

  • The secret was exposed somewhere — chat log, screenshot, accidental git commit, support ticket, screen-share
  • You suspect or know there was a breach
  • Someone with access leaves the team
  • A device that had the secret cached is lost or stolen
  • A vendor or service announces a breach affecting their side

Rotate as hygiene

  • Annually for high-value secrets (mailbox, AWS keys) — optional but reasonable
  • When the secret's TTL is about to expire (e.g. our GitHub PAT was 90-day)

Don't bother rotating

  • Strong (16+ char), uniquely generated secret
  • Stored only in a password manager / cloud secret store
  • Never exposed
  • Behind 2FA or scoped permissions

How to rotate each

1 + 2. hello@keelstack.uk mailbox + SMTP_PASS

These two are the same password. Rotate together.

  1. cPanel → Email Accounts → find hello@keelstack.uk → Manage → Set password
  2. Generate a strong 20-char password — PowerShell: 
    -join ((48..57) + (65..90) + (97..122) | Get-Random -Count 20 | ForEach-Object {[char]$_})
    Avoid $ for a quieter .env build (the build script escapes it correctly either way).
  3. Paste the new value as the cPanel mailbox password. Save.
  4. AWS Amplify Console → Environment variables → edit SMTP_PASS → paste same value → Save
  5. Amplify auto-rebuilds (~3 min)
  6. Verify: submit a test on https://www.keelstack.uk/seo with a real email — confirm the verification email arrives.

3. AWS IAM access keys

  1. AWS Console → IAM → Users → aerologue-ads-admin → Security credentials
  2. Under Access keys → Create access key → "Command Line Interface (CLI)" → save the new key + secret (secret shown only once)
  3. Update local CLI: 
    aws configure --profile aerologue-ads-dev
  4. Test: 
    aws sts get-caller-identity --profile aerologue-ads-dev
  5. Deactivate the old key, wait 24h of clean usage, then Delete.

4. GitHub PAT amplify-keelstack

  1. github.com/settings/tokens?type=beta → find amplify-keelstack
  2. Regenerate (preserves the name, gives a new token + TTL)
  3. Re-authorize Amplify only if it loses access (usually it doesn't — OAuth handles the install).

5. VERIFY_SECRET

⚠ before rotating Anyone with an unclicked verification email at the moment of rotation will get "this link has expired" when they click. Pick a low-traffic window.
  1. Generate 48-char random: 
    -join ((48..57) + (65..90) + (97..122) | Get-Random -Count 48 | ForEach-Object {[char]$_})
  2. Amplify Console → edit VERIFY_SECRET → paste → Save → rebuild (~3 min)
  3. Affected leads need to re-submit. Consider a one-day announcement if a campaign is live.

6. ANTHROPIC_API_KEY (when added)

  1. console.anthropic.com → API keys → Revoke the old → generate new → copy
  2. Amplify → edit env var → Save

7. RESEND_API_KEY (when added)

  1. resend.com/api-keys → revoke old → create new → copy
  2. Amplify → edit env var → Save

Don't do these things

  • Don't reuse passwords across cPanel, AWS, GitHub — one breach compromises everything
  • Don't store secrets in code, git, or screenshots
  • Don't share passwords over chat/SMS/email — use a password manager's secure share
  • Don't keep them in browser autofill on shared computers
  • Don't put production secrets in .env.example — that file gets committed

Tools to use

  • Password manager: 1Password, Bitwarden (free tier is great), KeePassXC
  • Random string generator: PowerShell one-liners above, openssl rand -base64 32, or 1Password's built-in
  • Burn inbox for verify-flow testing: tempmail-now.com, mailinator.com

After every rotation

  • Notify any team member who had the old secret
  • Update any cron jobs / CI runners that used the old credentials
  • Verify the affected service still works (form submit, deploy, API call)
  • Note the rotation date in your password manager
